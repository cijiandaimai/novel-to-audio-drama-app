# 酒馆 Human-in-the-loop 与世界线分支企划

更新时间：2026-07-08

## 1. 企划目标

把“酒馆”从普通 AI 角色聊天，升级为 **AI IP 世界预演系统**。

核心目标：

- 编剧可以随时介入 AI 推演。
- 编剧可以“夺舍”任意角色，亲自写关键台词、动作和反转。
- AI 能无缝承接人类编剧的语境、潜台词和角色意图。
- 剧情可以像 Git 一样保存、读档、分支和对比。
- 史官负责把每条世界线中已经发生的事实沉淀为史书。
- 最终把最好的世界线转入创作台，生成大纲、剧本、广播剧和影视化提案。

一句话定位：

> 玩家搭世界，AI 演角色，编剧随时夺舍修戏，史官记录史实，世界线管理不同剧情可能。

---

## 2. 为什么要做这个功能

当前酒馆已经具备：

- 角色卡
- 世界书
- 长期记忆
- 史官与史书
- API 模型选择
- 自动超级前置提示词
- 全部 TXT 导出
- 转入创作台

但还缺少两个专业创作工具必需能力：

### 2.1 人类编剧介入

AI 自动推演容易出现：

- 台词不够狠
- 爽点密度不够
- 关键反转写得太平
- 角色关系不够锋利
- 模型不敢让角色真正冲突

所以编剧必须能随时介入。

### 2.2 世界线分支管理

影视创作经常需要试错：

- 如果这个角色提前死亡会怎样？
- 如果秘密提前曝光会怎样？
- 如果男女主提前和解会不会失去张力？
- 如果反派不是坏人，而是债主，会不会更有戏？

这些都需要“平行宇宙”推演。

## 3. 核心功能一：人类编剧介入

### 3.1 发言身份

酒馆输入框增加“发言身份”：

```text
发言身份：
1. 玩家
2. 史官
3. 编剧夺舍
```

当选择“编剧夺舍”时，再选择要夺舍的角色：

```text
夺舍角色：
白泽 / 李七七 / 谢无痕 / 其他自定义角色
```

### 3.2 夺舍规则

当编剧夺舍某个角色输入内容时，系统应理解为：

```text
这不是普通玩家发言。
这是人类编剧临时扮演该角色说出的真实台词、动作或决定。
AI 必须承接，不得忽略、否定或重写。
```

示例：

```text
发言身份：编剧夺舍
夺舍角色：谢无痕
输入：
我烧掉藏魂阁，不是为了叛宗，是为了让里面的人魂不再被炼。
```

AI 后续回复应把这句话视为谢无痕亲口说出的信息，让其他角色自然反应。

### 3.3 数据结构

第一版消息结构：

```json
{
  "role": "user",
  "text": "我烧掉藏魂阁，不是为了叛宗，是为了让里面的人魂不再被炼。",
  "at": "2026-07-08T12:00:00.000Z",
  "writerMode": "possess",
  "possessedCharacterId": "xie-wuhen",
  "possessedCharacterName": "谢无痕",
  "source": "human-writer"
}
```

### 3.4 提示词规则

API 提示词中加入：

```text
【人类编剧介入】
本轮用户不是以普通玩家身份发言，而是由人类编剧夺舍「{角色名}」发言。
这句话视为「{角色名}」在当前世界线中的真实台词、行动或表达。
你必须承接它，不得否定、重写或忽略它。
如果它暴露秘密、改变关系或推动重大事件，请让当前回复体现影响。
```

### 3.5 OOC 冲突处理

如果夺舍内容明显违反原角色卡，不应直接阻止。

系统应按两种方式处理：

1. 临时表演：只在当前剧情中承接。
2. 写入角色状态：视为角色发生重大转变。

后续可增加提示：

```text
这次夺舍与原角色设定存在冲突，是否写入角色状态？
```

## 4. 核心功能二：世界线存档与分支

### 4.1 第一版先做手动存档

先不要直接做复杂 Git 树，第一版只做：

```text
保存当前世界线
读取世界线
另存为平行宇宙
删除存档
```

### 4.2 存档内容

每个存档应包含：

- 世界书
- 长期记忆
- 史书
- 角色卡
- 每个角色 API 模型
- 当前对话记录
- 史官身份状态
- 自动超级前置提示词配置
- 当前酒馆模式
- 当前选中角色

### 4.3 数据结构

```json
{
  "id": "branch-main-001",
  "name": "主世界线",
  "description": "谢无痕携玉牌入酒馆后的默认线",
  "createdAt": "2026-07-08T12:00:00.000Z",
  "updatedAt": "2026-07-08T12:30:00.000Z",
  "parentId": null,
  "forkedFromMessageId": null,
  "snapshot": {
    "characters": [],
    "sessions": {},
    "activeId": "baize",
    "world": "",
    "memory": "",
    "chronicle": "",
    "mode": "story",
    "engine": "api",
    "provider": "auto",
    "plotControl": {}
  }
}
```

### 4.4 平行宇宙

当用户点击“另开平行宇宙”时：

```text
复制当前所有状态
创建新的 branchId
用户输入新世界线名称
后续对话只影响这个分支
```

示例：

```text
主世界线
├─ 谢无痕提前死亡线
├─ 李七七身世提前曝光线
└─ 白泽黑化债主线
```

## 5. 核心功能三：分支对比

后续版本增加分支对比面板。

对比维度：

- 爽点密度
- 悬疑反转数量
- 情绪起伏
- 角色冲突强度
- 影视化潜力
- 广播剧可听性
- 是否出现 OOC
- 是否违反史书

第一版可先人工对比，第二版再接入 API 自动评分。

---

## 6. 和史官功能的关系

史官是世界线的“事实记录器”。

每条世界线都应该有自己的史书。

```text
主世界线：史书 A
谢无痕死亡线：史书 B
李七七黑化线：史书 C
```

规则：

- 史书只记录当前世界线已经发生的事实。
- 从某个节点分支后，旧史实继承，新事件独立发展。
- 已发生事实不能被后续 AI 随意推翻。
- 人类编剧夺舍产生的重大台词或行动，可由史官写入史书。

## 7. 推荐开发路线

### 阶段 1：编剧夺舍 MVP

目标：

- 增加发言身份选择。
- 支持玩家 / 史官 / 编剧夺舍。
- 编剧夺舍时选择角色。
- 消息记录夺舍角色。
- API 提示词承接夺舍内容。
- 对话气泡显示“编剧夺舍：角色名”。

验收标准：

- 用户可以选择“夺舍角色”。
- 输入内容后，对话中显示“编剧夺舍：谢无痕”。
- AI 回复能承接被夺舍角色的语境。
- 史官修史能识别该内容。

### 阶段 2：世界线手动存档

目标：

- 新增世界线管理面板。
- 支持保存当前世界线。
- 支持读取已有世界线。
- 支持删除存档。

验收标准：

- 保存后刷新页面仍能看到存档。
- 读取存档后，角色、对话、史书、世界书恢复一致。
- 不影响未追踪本地素材。

### 阶段 3：另开平行宇宙

目标：

- 基于当前世界线复制出新分支。
- 分支拥有独立对话、史书、记忆。
- UI 显示当前世界线名称。

验收标准：

- 主线和分支互不覆盖。
- 分支可继续对话。
- 分支史书独立更新。

### 阶段 4：史官审校 / OOC 检查

目标：

- 回复后检查是否违反角色卡、世界书、史书。
- 检查角色是否知道不该知道的信息。
- 第一版先只生成审校报告，不自动改写剧情。
- 后续再增加“按建议重写上一条回复”。

验收标准：

- 能指出 OOC 风险。
- 能指出史实冲突。
- 能给出修改建议。
- 能提示是否建议写入史书。

### 阶段 5：分支对比和评分

目标：

- 对比两条世界线。
- 输出爽点密度、反转数量、冲突强度、影视化潜力。
- 推荐更适合进入创作台的分支。

验收标准：

- 可以选择两个分支对比。
- API 生成简短评估。
- 能一键把推荐分支转入创作台。

```text

```

## 9. 存储规划

短期：

- 使用 localStorage 保存世界线快照。
- 先限制存档数量，例如最多 20 个。

中期：

- 迁移到 IndexedDB。
- 支持大量长篇对话和多个 IP 项目。

长期：

- 支持导出完整世界线包：

```text
角色卡
世界书
史书
角色状态
全部对话
分支树
创作台产物
```

## 10. 风险与注意事项

### 10.1 代码复杂度风险

这些功能会继续增加酒馆复杂度。

建议同步推进：

- 把酒馆逻辑逐步拆成模块。
- 封装状态读写。
- 给 Tavern 消息、角色、世界线加 JSDoc 类型。

### 10.2 OOC 风险

夺舍内容可能与角色卡冲突。

处理方式：

- 第一版先允许编剧强行介入。
- 后续增加“是否写入角色状态”的确认。

### 10.3 分支存储风险

如果直接复制全部状态，存储会变大。

处理方式：

- 第一版限制存档数量。
- 后续再做增量存储或 IndexedDB。
