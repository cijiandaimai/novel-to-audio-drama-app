# 白泽 IP 预演台创作流程完整说明

本文档梳理 App 从“导入小说”到“生成广播剧音频”的完整工作流，并说明 A 计划、B 计划、C 计划、直接音频生成入口、接口配置、产物保存和异常降级逻辑。

## 1. 产品定位

白泽 IP 预演台是一个 AI 影视预创作工具，目标是把小说、分场文本或已经写好的音频提示词，转换成可收听、可下载、可继续剪辑的广播剧音频。

App 本身不提供模型额度。用户需要到对应模型官网购买或开通额度，再把 API 信息填入 App。正式给他人安装使用时，推荐通过后端中转保存付费 Key，避免把 Key 暴露在 APK 里。

## 2. 创作入口

当前创作链路分为三类入口。

### 2.1 小说自动改编入口

适合用户只有小说正文、章节内容或长篇故事文本时使用。

输入内容：

- 作品标题。
- 小说文档或粘贴的小说正文。
- 可选的角色音色参考。
- API 配置。
- 网络配置。

输出内容：

- 项目资料包。
- 分场大纲。
- 台词方向。
- 剧本 1。
- 台词二次优化稿。
- 剧本 2。
- 分段音频提示词。
- 分段音频。
- 合成后的完整广播剧音频。

### 2.2 跳过编剧，直接生成音频入口

适合用户已经准备好了音频提示词，不想再让 App 走编剧流程时使用。

输入内容：

- 作品标题。
- JSON 数组格式的分段提示词，或普通文本分段提示词。
- 可选的角色音色参考。
- Doubao-Seed-Audio 1.0 音频配置。

处理方式：

- App 会把用户上传或粘贴的提示词整理成统一的片段数组。
- 每个片段包含 `id`、`title`、`durationSeconds`、`prompt`。
- 然后直接交给 Doubao-Seed-Audio 1.0 生成音频。
- 这个入口不依赖 GPT 或 Gemini。

### 2.3 午夜猫娘入口

这是受门禁保护的特殊改写入口。用户需要通过本地题库考核后才会显示或启用。它主要用于把已有文本交给 Grok 进行特定风格改写。

注意：

- 该入口独立于主创作流程。
- Grok API 需要单独配置。
- 门禁状态保存在本地。
- 连续失败达到限制后，本地会隐藏入口。

## 3. A 计划：最佳多模型流水线

A 计划是当前设计的最佳质量方案，使用不同模型负责自己最擅长的环节。

### 3.1 A 计划适用条件

用户已经配置好：

- GPT 或 OpenAI 兼容文本模型。
- Gemini 文本模型。
- 豆包文本模型。
- Doubao-Seed-Audio 1.0 音频模型。

其中豆包文本模型不仅用于中文台词润色，也作为 B 计划的兜底模型。

### 3.2 A 计划完整流程

1. 用户导入小说

App 读取小说内容，清理换行和空白，并按章节标题或长度自动拆分。长文本不会在每一步反复整本传递，而是提取开头样本、中段样本、结尾样本、字符数和拆分段数，组成小说上下文。

2. GPT 生成项目资料包

GPT 作为总编剧和制作统筹，负责把小说转成广播剧项目资料包。

这一阶段产物包括：

- 故事骨架。
- 角色小传。
- 人物关系。
- 场景列表。
- 广播剧改编策略。
- 不得改动的关键设定。

3. Gemini 扩展分场大纲

Gemini 负责长文本理解和场景扩展。

这一阶段重点是：

- 增强戏剧冲突。
- 补足场景动作。
- 标注每场情绪曲线。
- 保持原著核心剧情一致。

4. 豆包文本模型优化对白方向

豆包文本模型负责中文表达和广播剧听感。

这一阶段产物包括：

- 角色语气规则。
- 台词风格禁忌。
- 每场关键对白建议。
- 更适合中文广播剧的口语表达方向。

5. GPT 生成剧本 1

GPT 汇总前面三个阶段的内容，生成第一版广播剧剧本。

剧本 1 必须包含：

- 场次编号。
- 场景空间。
- 角色。
- 旁白。
- 对白。
- 音效。
- 音乐。
- 情绪提示。
- 预计时长。

6. 豆包文本模型二次优化台词

豆包只优化台词和表演语气。

这一阶段的限制是：

- 不改剧情。
- 不改场次。
- 不删除音效和音乐标注。
- 不新增会破坏原著设定的内容。

7. GPT 生成剧本 2

GPT 根据二次优化稿，生成最终录制版剧本。

剧本 2 更接近音频制作稿，需要保留：

- 清晰分段。
- 角色声线。
- 音效时间线。
- 音乐设计。
- 混音要求。
- 可直接进入提示词生成的结构。

8. GPT 生成影视级音频提示词

GPT 读取 `AI影视级音频提示词写作指导.md` 的规则，把剧本 2 拆成 Doubao-Seed-Audio 1.0 可用的分段提示词。

每段提示词建议控制在 30 到 90 秒。复杂多人戏应拆短。

标准片段结构：

```json
{
  "id": "scene-01",
  "title": "片段名",
  "durationSeconds": 60,
  "prompt": "完整音频提示词"
}
```

提示词应包含：

- 音频类型。
- 时长。
- 整体情绪。
- 场景空间。
- 音乐设计。
- 环境声。
- 角色声线。
- 对白内容。
- 表演要求。
- 音效时间线。
- 混音要求。
- 禁止事项。

9. Doubao-Seed-Audio 1.0 分段生成音频

App 把每段提示词逐条提交给 Doubao-Seed-Audio 1.0。

请求中会包含：

- 模型 ID。
- 文本提示词。
- 目标时长。
- 音频格式。
- 采样率。
- 可选的角色音色参考。

10. 后台合成完整广播剧

如果走后端中转，服务器会把每段音频保存到本次任务目录，并尝试合成 `final-broadcast-drama.wav`。

如果走 App 内直连，前端会保存每段音频的链接或 dataUrl，并返回可播放的音频结果。

11. 写入历史记录

完成后保存本次任务信息，方便用户在历史页查看、下载或继续处理。

## 4. B 计划：中国大陆用户优先的兜底流程

B 计划用于 GPT 或 Gemini 无法连接、未配置、额度不足、请求失败时，自动改用豆包文本模型继续完成创作。

### 4.1 B 计划适用条件

至少需要配置：

- 豆包文本模型 API Key。
- 豆包文本模型 ID。
- 豆包文本模型接口地址。
- Doubao-Seed-Audio 1.0 音频接口。

推荐豆包文本配置：

```text
接口地址：https://ark.cn-beijing.volces.com/api/v3/chat/completions
模型 ID：doubao-seed-2-1-pro-260628
```

推荐音频配置：

```text
接口地址：https://openspeech.bytedance.com/api/v3/tts/create
模型 ID：seed-audio-1.0
```

实际模型 ID 以用户在火山方舟控制台开通的模型或接入点为准。

### 4.2 B 计划触发条件

以下情况会触发 B 计划：

- GPT 没有填写 API Key。
- GPT 没有填写模型 ID。
- GPT 请求失败。
- GPT 接口不可达。
- Gemini 没有填写 API Key。
- Gemini 没有填写模型 ID。
- Gemini 请求失败。
- Gemini 接口不可达。

### 4.3 B 计划执行方式

B 计划不会改变流程结构，只会把原本由 GPT 或 Gemini 负责的文本阶段替换成豆包文本模型执行。

也就是说：

- “GPT 生成项目资料包”会改用豆包文本模型完成。
- “Gemini 扩展分场大纲”会改用豆包文本模型完成。
- “GPT 生成剧本 1”会改用豆包文本模型完成。
- “GPT 生成剧本 2”会改用豆包文本模型完成。
- “GPT 生成影视级音频提示词”会改用豆包文本模型完成。
- 原本就由豆包负责的台词优化阶段保持不变。
- 音频生成仍然由 Doubao-Seed-Audio 1.0 执行。

### 4.4 B 计划的优点

- 对中国大陆网络环境更友好。
- 用户只需要优先配置火山方舟相关模型。
- 流程不会因为 GPT 或 Gemini 不可用而直接中断。
- 仍然能生成完整剧本、提示词和音频。

### 4.5 B 计划的质量差异

B 计划更稳定，但创作分工没有 A 计划细。

可能的差异：

- 长篇复杂小说的结构分析能力可能不如 A 计划。
- 场景扩写的风格可能更统一，但变化性略少。
- 提示词生成质量取决于豆包文本模型对 JSON 结构的稳定输出。

建议：

- 普通用户默认走 B 计划即可。
- 对质量要求高、并且网络和额度都稳定时，再启用 A 计划。

## 5. C 计划：国产多模型增强流程

C 计划用于在中国大陆使用环境下提升文本创作质量。它不是替代 B 计划，而是在豆包稳定底座上加入千问 Qwen 和月之暗面 Kimi，让长文理解、结构化剧本和提示词输出更细。

### 5.1 C 计划适用条件

建议至少配置：

- 豆包文本模型。
- 千问 Qwen。
- Kimi。
- Doubao-Seed-Audio 1.0。

如果只配置了千问或 Kimi 其中一个，App 也会尽量启用 C 计划，并用豆包文本模型兜底。

推荐千问配置：

```text
接口地址：https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions
模型 ID：qwen-plus
```

推荐 Kimi 配置：

```text
接口地址：https://api.moonshot.cn/v1/chat/completions
模型 ID：以 Kimi API 平台当前可用模型为准
```

### 5.2 C 计划模型分工

| 环节 | 推荐模型 | 职责 |
|---|---|---|
| 项目资料包 | Kimi 优先，千问兜底 | 长篇小说理解、人物关系、伏笔、章节脉络 |
| 分场大纲 | Kimi 优先，千问兜底 | 长上下文扩展、冲突补强、情绪曲线 |
| 中文台词方向 | 豆包文本 | 中文对白、口语节奏、角色语气 |
| 剧本 1 | 千问优先，Kimi 兜底 | 结构化剧本、场次表、旁白和音效格式 |
| 台词二次优化 | 豆包文本 | 只改台词和表演语气，不改剧情 |
| 剧本 2 | 千问优先，Kimi 兜底 | 最终录制版、时间线、混音要求 |
| 音频提示词 | 千问优先，豆包兜底 | 稳定 JSON、分段提示词、格式约束 |
| 音频生成 | Doubao-Seed-Audio 1.0 | 分段生成音频 |

### 5.3 自动推荐规则

App 的“自动推荐”会按下面顺序选择方案：

1. GPT、Gemini、豆包文本都可用时，选择 A 计划。
2. 没有完整 A 计划，但千问或 Kimi 可用时，选择 C 计划。
3. 只有豆包文本可用时，选择 B 计划。
4. 只有豆包音频可用时，建议使用“直接用提示词生成音频”入口。

### 5.4 C 计划与 B 计划的区别

B 计划是稳定优先：尽量用豆包文本模型完成所有文本环节。

C 计划是国产增强：把长文理解交给 Kimi，把结构化输出交给千问，把中文台词继续交给豆包。它比 B 计划更适合长篇小说和复杂广播剧，但配置项会多一些。

## 6. 演示模式

如果某个模型没有配置，并且也没有可用的 B 计划，App 会在部分阶段使用演示内容。

演示模式的作用：

- 让用户不用买 API 也能先体验完整流程。
- 方便检查界面、历史记录、播放器和下载链路。
- 生成静音占位 WAV，避免流程中断。

演示模式不适合正式创作，因为它不会调用真实模型生成高质量内容。

## 8. 网络与后端中转

App 有两种运行方式。

### 8.1 App 内直连

适合自己测试。

特点：

- 配置简单。
- API Key 存在本地。
- Android APK 被反编译时可能暴露 Key。
- 网络环境依赖用户手机本身。

### 8.2 后端中转

适合正式分发给他人使用。

特点：

- 付费 Key 放在服务器环境变量里。
- App 只填写后端中转地址。
- 可以统一做限流、鉴权、日志和错误处理。
- 对中国大陆用户可以使用更稳定的服务器网络。

如果填写了后端中转地址，创作、直接音频生成、历史记录和联网诊断都会优先请求服务器接口。中转失败时不会自动回退成本机直连，以免付费 Key 暴露在 APK 侧。

## 9. 角色音色参考

用户可以上传或录制自己的声音作为参考音色。

当前处理方式：

- App 将音色参考保存为本地数据。
- 调用音频接口时最多取前 3 条参考音色。
- 请求音频接口时把参考音频转成 base64 放入 `references`。

注意：

- 参考音色应使用用户自己拥有权利的声音。
- 不应上传真实演员、主播、公众人物或未授权他人的声音。
- 提示词中也应要求使用原创合成声线，不模仿真实人物。

## 10. 音频生成与合成

音频生成按片段逐段执行。

好处：

- 某一段失败时只影响当前片段。
- 长剧本更容易控制时长。
- 后续剪辑时可以按段调整。
- 方便保存每一段的提示词和音频结果。

后端合成逻辑：

- 优先保存每个片段的 WAV 文件。
- 如果片段是标准 WAV，会把多个 WAV 串接成一个完整音频。
- 输出文件名为 `final-broadcast-drama.wav`。

当前合成方式偏轻量。后续如果要做更专业的响度统一、淡入淡出、跨格式转码和多轨混音，建议接入 FFmpeg 或专门的音频处理服务。

## 11. 历史记录与产物

后端中转模式下，每次创作会生成一个任务目录。

主要产物：

```text
01-project-bible.md
02-expanded-treatment.md
03-dialogue-brief.md
04-script-1.md
05-dialogue-pass.md
06-script-2.md
07-audio-prompts.json
manifest.json
stage-log.md
final-broadcast-drama.wav
```

其中：

- `manifest.json` 是任务清单。
- `stage-log.md` 是完整阶段日志。
- `06-script-2.md` 是最终录制版剧本。
- `07-audio-prompts.json` 是可复用的分段音频提示词。
- `final-broadcast-drama.wav` 是合成后的完整音频。

App 内直连模式下，因为没有本地 Node 文件服务，产物主要以浏览器对象链接或 dataUrl 的方式返回，适合移动端即时播放和下载。

## 12. 播放器与音箱页

音箱页承担播放器、播放列表、歌词和内容归档入口。

播放器支持：

- 导入外部音频。
- 导入 LRC 或 KRC 歌词。
- 判断逐行歌词或逐字歌词。
- 小窗歌词跟随播放进度。
- 全屏歌词模式。
- 双击退出全屏。
- 记录播放列表。
- 记录上次导入路径。
- 扫描文件夹导入音频和歌词。
- 播放记录文档，用于快速恢复播放状态。

逐字歌词显示时，当前字会跟随播放进度高亮。普通 LRC 则按行滚动和高亮。

## 13. 剪辑模块

剪辑页用于对生成音频或外部音频做基础后期。

当前目标：

- 移动端单屏操作。
- 双轨导入。
- 长按拖动音频片段。
- 选区剪辑。
- 撤销和重做。
- 麦克风录音。
- 双轨混音导出。

当前剪辑模块是轻量实现，适合简单拼接、预览和混音。更专业的降噪、人声背景分离、响度分析和复杂多轨工程，后续建议引入成熟音频引擎或在服务端使用 FFmpeg、Demucs 等工具处理。

## 14. 错误处理逻辑

### 14.1 文本模型错误

GPT 或 Gemini 错误：

- 如果豆包文本模型配置完整，自动切换 B 计划。
- 如果豆包文本模型也不可用，真实请求会失败或返回演示内容，具体取决于当前阶段是否有 mock 内容。

豆包文本错误：

- 台词优化阶段会失败或回到演示内容。
- B 计划无法继续兜底，因为 B 计划本身依赖豆包文本模型。

### 14.2 音频模型错误

Doubao-Seed-Audio 1.0 错误：

- 当前片段会生成静音占位 WAV。
- 结果中会保留错误信息。
- 其他片段继续执行。

### 14.3 网络错误

网络配置支持：

- 超时时间。
- 重试次数。
- 后端中转地址。

中国大陆用户建议优先使用火山方舟模型和后端中转，减少跨境接口不可达导致的失败。

## 15. 推荐使用方式

### 15.1 普通中国大陆用户

推荐默认 B 计划：

1. 只配置豆包文本模型。
2. 配置 Doubao-Seed-Audio 1.0。
3. 不填 GPT 和 Gemini。
4. 直接上传小说生成广播剧。

这样最稳定，配置成本也最低。

### 15.2 高质量创作用户

推荐 A 计划：

1. 配置 GPT。
2. 配置 Gemini。
3. 配置豆包文本模型。
4. 配置 Doubao-Seed-Audio 1.0。
5. 使用后端中转保护 Key。

这样模型分工更细，剧本结构、扩写和中文台词质量更容易拉开。

### 15.3 中国大陆增强创作用户

推荐 C 计划：

1. 配置豆包文本模型。
2. 配置千问 Qwen。
3. 配置 Kimi。
4. 配置 Doubao-Seed-Audio 1.0。
5. 创作方案选择“自动推荐”或“C 计划”。

这样比纯 B 计划更适合长篇小说，同时比 A 计划更贴近中国大陆网络和模型购买环境。

### 15.4 已有提示词用户

推荐直接音频入口：

1. 跳过编剧。
2. 上传 JSON 或普通分段提示词。
3. 配置 Doubao-Seed-Audio 1.0。
4. 直接生成音频。

这样速度最快，也最省文本模型费用。

## 16. 一句话总结

A 计划是“国际多模型分工协作，质量优先”；B 计划是“豆包文本模型接管 GPT/Gemini，稳定优先”；C 计划是“千问 + Kimi + 豆包的国产多模型增强”。用户在中国大陆环境下可以先用 B 计划跑通完整流程，再按需要配置千问和 Kimi 切换到 C 计划，最后有条件时再用 A 计划追求更高质量。
